# Datasheet — BBO Capstone Query Dataset

**Author:** Nikolas Karefyllidis, PhD
**Last updated:** April 2026 (after Round 10)

---

## Motivation

**Purpose.** This dataset was created to support the ICL Black-Box Optimisation (BBO) Capstone Project — a ten-round sequential optimisation challenge modelled on the NeurIPS 2020 BBO Competition. Its primary task is to train and evaluate a Gaussian Process surrogate that proposes the next query point for each of eight unknown objective functions, maximising the returned scalar output within a strict per-round evaluation budget.

**Creator.** Nikolas Karefyllidis, PhD student / ML practitioner, as a required deliverable for the Imperial College London PCMLAI programme (Module 21). The challenge portal and initial data were provided by ICL course staff.

**Funding.** No external funding. Created as coursework.

---

**What the instances represent.** Each instance is a single function evaluation: an input vector **x** ∈ [0, 1]^d submitted to the challenge portal and the scalar output y returned by the portal's oracle.

**Format.** Each function's observations are stored as a single CSV file (`data/problems/function_N/observations.csv`) with columns `x1, x2, …, xd, y`. The initial warm-start data (10 points per function) is stored separately as read-only `.npy` files in `initial_data/function_N/`.

**Missing data.** No missing values. Every submitted x received a y from the portal. F6 contains three evaluations at the identical input vector (rounds 8, 9, 10), retained deliberately to capture oracle noise (outputs: −0.237, −0.188, −0.161 respectively — a spread of ~0.08 suggesting mild stochasticity in the oracle).

**Confidentiality.** None. No personally identifiable information. The oracle functions are unknown to the student but are not confidential.

## Collection Process

**How queries were generated.** Queries were generated by a GP-based Bayesian Optimisation pipeline implemented in per-function Jupyter notebooks (`notebooks/function_N_*.ipynb`). The pipeline:
1. Fits three GP kernels (RBF, Matérn ν=1.5, RBF+WhiteKernel) to all prior observations.
2. Selects the best kernel by log-marginal likelihood (LML).
3. Optimises an acquisition function (EI, PI, or UCB) over a Sobol/LHS candidate set of 2^15–2^18 points.
4. Returns the candidate with the highest acquisition value, subject to minimum-distance and boundary constraints.

The first 10 observations per function were provided as initial data by ICL (not generated by this pipeline). Subsequent observations (rounds 1–10, one per function per round) were generated by the pipeline above.

**Sampling strategy.** Sequential model-based optimisation: each new query is conditioned on all prior (x, y) pairs. Early rounds (~rounds 1–3) used higher exploration weighting (UCB with κ = 2–3, or EI with ξ = 0.05–0.1). Later rounds (~rounds 7–10) shifted toward exploitation (EI with ξ = 0.01, UCB with κ = 1–2) as the surrogate became more confident near promising regions. An ensemble of acquisition functions (centroid when EI/PI/UCB disagree by > L2 threshold of 0.15) was used from round 4 onward.

**Time frame.** January 27, 2026 (Round 1) to April 9, 2026 (Round 10). One submission per function per week; ten rounds total.

## Preprocessing / Cleaning / Labelling

**Transformations applied.** Output warping (HEBO-inspired) is applied to y before GP fitting for three functions where the raw output distribution is heavily skewed or multi-scale:
- F1 (radiation detection): log transform — y spans [−0.006, 0.670], mostly near zero with a single large value.
- F5 (chemical yield): log transform — y spans [0.11, 7493.9], roughly four orders of magnitude.
- F7 (hyperparameter tuning): log transform — y spans [0.003, 2.797].
- F2, F3, F4, F6, F8: no transform (raw y used).

The inverse transform is applied when reporting results and best-observed values; raw y is always stored in `observations.csv`.

**Duplicate-x handling.** From round 11 onward, if the same x is evaluated in multiple rounds (as occurred for F6), all observations are retained as separate rows rather than overwriting. This allows the GP WhiteKernel to estimate oracle noise variance from the spread. For rounds 1–10, F6's three duplicate rows were reconstructed from portal feedback emails and appended after the fact.

**Raw data preservation.** Yes — `initial_data/` is read-only and never modified. All appended (x, y) pairs are stored verbatim as received from the portal; no values have been corrected, filtered, or imputed.

## Uses

**Intended uses.**
- Training and evaluating GP surrogates for sequential black-box optimisation (maximisation).
- Benchmarking acquisition functions (EI, PI, UCB, ensemble) on real oracle feedback.
- Teaching and demonstrating Bayesian optimisation workflows, exploration–exploitation trade-offs, and surrogate diagnostics.
- Reproducing the query strategy described in this project for portfolio or academic review.

**Potential limitations for other uses.** The dataset is small (20–48 points per function) and domain-specific. The oracle functions are unknown, so ground truth performance cannot be assessed. Using this dataset to train a surrogate for a different optimisation problem requires caution — the query distribution is biased toward regions that appeared promising under this specific GP/acquisition strategy, not toward uniform coverage.

**Inappropriate uses.**
- Do not use this dataset to make inferences about the true functional form of the oracle — the functions are unknown and the observations are too sparse for reliable inference.
- Do not use observations from one function as training data for a surrogate of a different function — the eight functions are unrelated.
- The dataset should not be used for any commercial or medical optimisation task; it was created solely for educational purposes.
